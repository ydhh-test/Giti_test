# 项目代码规范（Agent 短版）

## 1. 架构边界

- API 层负责接口边界、输入输出组织、协议适配。
- 数据模型层负责数据结构定义与结构校验。
- 核心业务流程必须放在 `core`、`nodes`、`rules` 等下层。
- 不要把 API 边界逻辑、核心业务流程逻辑、数据模型职责混写在同一层。

---

## 2. 数据模型规范

- 这里的“模型”专指数据结构体与数据模型，不指 AI 模型、算法模型或训练模型。
- 新的数据模型统一使用 **pydantic**。
- 所有数据模型字段必须显式声明类型。
- 可变默认值必须使用：
  - `Field(default_factory=list)`
  - `Field(default_factory=dict)`
- 不允许使用共享可变默认值。
- 优先使用明确的结构化模型，不长期依赖 `dict[str, Any]`。

---

## 3. API 规范

- API 输入输出边界优先使用显式模型。
- 除明确的边界兼容或占位场景外，不使用 `Any`。
- API 层保持薄层，不承载复杂业务流程。
- 占位 API 必须保持占位性质，不伪装成完整实现。

---

## 4. 类型规范

- 优先强类型建模。
- `Any` 只允许用于：
  - 结构确实未知
  - 占位阶段暂未建模
  - 边界兼容必须保留宽泛输入
- `dict[str, Any]` 只允许作为暂未结构化的过渡字段。
- 一旦结构稳定，应升级为明确模型。

---

## 5. 文件组织规范

- 数据模型放在 `src/models/`
- API 入口放在 `src/api/`
- 单元测试放在 `tests/unittests/`
- 集成测试放在 `tests/integrations/`
- 文件名应体现业务对象或职责，不使用模糊命名。

---

## 6. 依赖与版本规范

- 新增第三方依赖时，必须同步更新 `requirements.txt`。
- 不允许使用未声明依赖。
- `version.json` 放在项目根目录。
- `version.json` 只允许由架构师修改。
- 研发工程师的日常 PR 默认不修改 `version.json`，除非该变更已被明确认定为版本级变更并由架构师要求更新。

---

## 7. 错误处理规范

- 错误码设计应清晰分层。
- 错误信息必须能帮助定位问题。
- 错误处理逻辑应与核心业务逻辑分离。
- 不允许使用无定位价值的模糊错误提示。

---

## 8. 变更控制

- 优先做最小改动，不做无关重构。
- 不在同一任务中顺手扩展未被要求的功能。
- 占位文件的改造应保持占位语义，除非任务明确要求补完整实现。
- 没有明确复用价值时，不引入额外抽象层。

---

## 9. 测试规范

- 测试应优先明确输入与预期输出。
- 对结构化对象，优先比较整体结果或关键结构。
- 除非目标是验证单个字段，否则不推荐使用大量逐字段断言表达整体预期。

---

## 10. 函数边界校验

- 每个函数应对自己的入参边界负责。
- 公共函数、对外函数、跨层调用函数必须显式检查输入是否满足约定。
- 不在内部重复校验已经由上层稳定保证的条件。

---

## 11. 文档一致性

- 有设计文档时，以设计文档为准。
- 文档与代码现状冲突时，必须显式说明偏离点。
- 不允许静默偏离设计文档。

---

## 12. 目标状态

- 项目数据模型层统一使用 **pydantic**。
- API 输入输出全面使用显式模型。
- API 层不再依赖宽泛的 `Any`。
- `requirements.txt` 作为依赖声明唯一来源。
- `version.json` 由架构师维护，并与协议/接口版本保持同步。
- 项目整体优先使用结构化模型，而不是临时拼装的松散对象。
