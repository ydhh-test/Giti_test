# 项目架构与代码规范

## 1. 适用范围

本文档定义项目级别的架构约束与代码规范，适用于本项目后续开发、重构、评审与自动化执行。

默认情况下，所有新增代码和修改代码都应遵守本文档。  
如需偏离，必须明确说明偏离原因与影响范围。

---

## 2. 优先级定义

- **L0 不可违反**：必须严格遵守
- **L1 强制标准**：新代码默认必须遵守
- **L2 推荐标准**：没有明确理由时应遵守
- **L3 目标状态**：项目整体最终应达到的统一标准

---

## 3. L0 不可违反

### 3.1 架构边界

- API 层负责接口边界、输入输出组织、协议适配。
- 数据模型层负责数据结构定义与结构校验。
- 核心业务流程必须放在对应下层，如 `core`、`nodes`、`rules`。
- 不允许在同一层混写以下职责：
  - API 边界逻辑
  - 核心业务流程逻辑
  - 数据模型职责

### 3.2 依赖管理

- 新增任何第三方依赖时，必须同步更新 `requirements.txt`。
- 不允许使用未声明在依赖文件中的第三方包。

### 3.3 版本管理

- 协议、接口或版本相关变更，必须同步更新 `version.json`。
- `version.json` 只允许由架构师修改。
- 研发工程师的日常 PR 默认不修改 `version.json`，除非该变更已经被明确认定为版本级变更并由架构师要求更新。
- 不允许存在代码行为已变更但版本文件未按架构决策同步更新的情况。

---

## 4. L1 强制标准

### 4.1 数据模型标准

- 本文中的“模型”专指数据结构体与数据模型，不指 AI 模型、算法模型或训练模型。
- 新的数据模型统一使用 **pydantic**。
- 所有数据模型字段必须显式声明类型。
- 可变默认值必须使用：
  - `Field(default_factory=list)`
  - `Field(default_factory=dict)`
- 不允许使用共享可变默认值。
- 数据模型应优先表达明确结构，不应长期依赖松散的无边界对象。

### 4.2 API 标准

- API 输入输出边界应优先使用显式模型。
- 除明确的边界兼容或占位场景外，不应使用 `Any`。
- API 层应保持薄层，不直接承载复杂业务流程。
- 占位 API 实现必须保持占位性质，不应伪装成完整实现。

### 4.3 类型标准

- 优先使用强类型建模。
- `Any` 仅允许用于以下场景：
  - 结构确实未知
  - 占位阶段暂未建模
  - 边界兼容必须保留宽泛输入
- `dict[str, Any]` 仅允许作为暂未结构化的过渡字段使用。
- 一旦字段结构稳定，应将松散容器升级为明确模型。

### 4.4 文件组织标准

- 数据模型放在 `src/models/`
- API 入口放在 `src/api/`
- 单元测试放在 `tests/unittests/`
- 集成测试放在 `tests/integrations/`
- 文件名应体现业务对象或职责，不使用模糊命名。

### 4.6 函数边界校验标准

- 每个函数应对其入参边界负责。
- 公共函数、对外函数、跨层调用函数必须显式检查输入是否满足约定。
- 入参检查应与函数所在层的职责一致：
  - API 层检查接口边界输入
  - 数据模型层检查结构合法性
  - 下层业务函数检查其业务前置条件
- 不要求在内部重复校验已经由上层稳定保证且不会变化的条件。
- 类型错误与调用方式错误应优先视为编程错误；业务输入不合法应按业务错误处理。

### 4.5 错误处理标准

- 错误码设计应保持清晰分层。
- 错误信息必须能够帮助定位问题。
- 错误处理逻辑应与核心业务逻辑分离。
- 不允许用无定位价值的模糊错误提示替代真实错误信息。

## 5. L2 推荐标准

### 5.1 变更范围控制

- 优先做最小改动，不做无关重构。
- 不在同一任务中顺手扩展未被要求的功能。
- 占位文件的改造应保持占位语义，除非任务明确要求补完整实现。

### 5.2 文档一致性

- 有设计文档时，应以设计文档为准。
- 文档与代码现状冲突时，必须显式记录偏离点。
- 不允许静默偏离设计文档。

### 5.3 实现复杂度控制

- 优先先解决明确的类型边界问题，再做更深层结构化。
- 没有明确复用价值时，不引入额外抽象层。
- 代码复杂度应与问题规模匹配。

### 5.5 测试写法规范

- 测试应优先明确输入与预期输出。
- 对结构化对象的测试，优先比较整体结果或关键结构，不优先采用逐字段零散断言。
- 除非测试目标就是验证某个单独字段，否则不推荐使用大量逐字段断言来表达整体预期。
- 推荐做法：
  - 明确构造输入
  - 明确给出预期输出
  - 使用整体比较、结构比较或关键结果比较
- 不推荐使用大量零散断言替代完整预期，例如：
  - `assert meta.width == 512`
  - `assert meta.height == 512`
  - `assert meta.channel == 3`
- 对 pydantic 数据模型，优先比较对象整体、`model_dump()` 结果或明确的预期字典。

### 6.1 数据模型层目标状态

- 项目数据模型层统一使用 **pydantic**。
- 核心业务对象使用明确的结构化模型表达。
- `dict[str, Any]` 仅保留在真正的边界占位场景中。
- 数据模型层具备清晰的字段定义、默认值规则和结构校验能力。

### 6.2 API 层目标状态

- API 输入输出全面使用显式模型。
- API 层不再依赖宽泛的 `Any` 类型签名。
- API 层只承担接口边界职责，不承载核心流程实现。

### 6.3 依赖与版本目标状态

- `requirements.txt` 作为依赖声明的单一事实来源。
- `version.json` 仅由架构师维护，并与协议或接口版本保持同步。
- 代码中的实际依赖与依赖声明保持一致。

### 6.4 项目一致性目标状态

- 数据模型定义、API 边界、类型注解、默认值写法、文件组织在全项目范围内保持一致。
- 占位代码保持明确可识别的占位性质。
- 项目整体优先使用结构化模型，而不是临时拼装的松散对象。

---

## 7. 数据模型与 API 设计补充规则

### 7.1 数据模型设计

- 先定义结构，再补充校验。
- 先完成显式字段建模，再考虑是否需要拆分子模型。
- 公共字段是否抽取基类，应以明确复用价值为前提。
- 不为了“看起来完整”而提前引入不稳定字段。

### 7.2 API 设计

- API 层优先接收和返回稳定的业务模型。
- 如果 API 暂时是占位实现，应先统一类型签名，再决定是否补业务逻辑。
- 不应为了局部便利长期保留 `Any` 签名。

---

## 8. 依赖与版本文件规则

### 8.1 `requirements.txt`

- 项目第三方依赖统一声明在 `requirements.txt`
- 只写实际实现中真实使用到的第三方包
- 新增依赖时，必须与代码变更一起提交

### 8.2 `version.json`

- 放置于项目根目录
- 用于表达接口、协议或项目版本信息
- 仅由架构师维护
- 当版本相关行为变化时，必须由架构决策后同步维护

---

## 9. 代码评审关注点

评审时应重点检查以下问题：

- 是否把业务流程逻辑错误地写进 API 层或数据模型层
- 是否引入了未声明依赖
- 是否仍然在不必要地使用 `Any`
- 是否使用了共享可变默认值
- 是否新增了无边界的 `dict[str, Any]`
- 是否做了无关任务范围的顺手重构
- 是否与设计文档不一致且未说明偏离原因
- 是否未经架构决策修改 `version.json`

---

## 10. 最终原则

- 先明确边界，再写实现
- 先结构清晰，再功能扩展
- 先统一技术栈，再统一业务结构
- 代码应服务于长期一致性，而不是局部临时便利
